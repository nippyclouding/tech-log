# Tech Log 운영 가이드

## 1. 문서 목적

이 문서는 `https://techlog.site` 서비스를 현재 운영 구성으로 유지하기 위해 알아야 할 사항을 정리한다. 배포를 처음 구성한 과정과 상세 명령 기록은 [`deploy.md`](./deploy.md)를 참고한다.

기준일은 `2026-05-27`이며, 현재 구조는 RDS, S3, CloudFront, CloudWatch, Flyway를 사용하지 않는 EC2 단일 서버 구성이다.

## 2. 현재 운영 구성

```text
사용자
  -> techlog.site / Elastic IP
  -> EC2 t3.small, EBS gp3 25 GiB
       -> nginx: React 정적 파일, HTTPS, reverse proxy
       -> backend: Spring Boot API, 관리자 콘솔, 이메일 발송
       -> postgres: 애플리케이션 DB
       -> certbot: Let's Encrypt 인증서 자동 갱신 점검
```

| 항목 | 현재 값/방식 |
| --- | --- |
| 도메인 | `techlog.site` |
| HTTPS | Nginx + Let's Encrypt Certbot |
| EC2 운영 경로 | `/home/ubuntu/tech-log` |
| 애플리케이션 배포 | GitHub Actions build -> GHCR push -> EC2 pull/run |
| DB | EC2 내부 `postgres:16-alpine` 컨테이너 |
| 이미지 파일 | Docker volume `tech-log-images` |
| 관리자 인증 | `ADMINS` 테이블의 BCrypt 암호 해시 기반 Spring Security 로그인 |
| 구독 메일 | Gmail SMTP 앱 비밀번호 기반 발송 |

인터넷에 공개되는 포트는 Nginx의 `80`, `443`이다. Backend `8080`과 PostgreSQL `5432`는 Docker 네트워크 내부에서만 사용한다. SSH `22`는 GitHub Actions SSH 배포 방식을 사용하는 동안 접근 정책을 별도로 관리해야 하며, 불필요하게 전체 공개하지 않는 것이 원칙이다.

## 3. 운영 접근 경로

| 기능 | 주소 |
| --- | --- |
| 공개 사이트 | `https://techlog.site` |
| 관리자 콘솔 | `https://techlog.site/admin` (`/admin-console`은 호환 redirect) |
| 이메일 구독 확인 | 메일 안의 `/api/subscriptions/confirm?token=...` 링크 |

최초 전환 시 `ADMINS` 테이블이 비어 있으면 EC2 `.env.prod`의 `ADMIN_CONSOLE_USERNAME`, `ADMIN_CONSOLE_PASSWORD` 값으로 초기 관리자 계정이 한 번 생성된다. 이후 로그인 인증은 DB의 BCrypt 암호 해시를 사용하며, `.env.prod` 값을 바꾸는 것만으로 기존 관리자 비밀번호가 변경되지는 않는다. 비밀번호나 Gmail 앱 비밀번호를 Git 저장소, 문서, GitHub issue에 기록하지 않는다.

```bash
cd /home/ubuntu/tech-log
docker compose --env-file .env.prod exec -T postgres /bin/sh -c \
  'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "SELECT username, role, is_active, failed_login_attempts, locked_until, last_login_at FROM \"ADMINS\";"'
```

관리자 계정에는 `username`, `password_hash`, `display_name`, `role`, `is_active`, 실패 횟수와 잠금 만료 시각, 마지막 로그인 시각, 비밀번호 변경·생성·수정 시각을 저장한다. 로그인 실패가 5회 누적되면 15분 동안 잠긴다. 현재는 계정 관리 화면과 비밀번호 변경 API는 아직 없으므로, 운영에서 비밀번호 변경 기능을 추가할 때에는 BCrypt 갱신과 감사 로그를 함께 구현해야 한다.

## 4. 일상 상태 확인

### 4.1 서비스 확인

```bash
cd /home/ubuntu/tech-log
docker compose --env-file .env.prod ps
curl -I https://techlog.site
curl -i 'https://techlog.site/api/posts?page=0&size=1'
```

확인 기준:

| 확인 대상 | 정상 기준 |
| --- | --- |
| `nginx` | `Up` 상태, HTTPS 페이지가 `200` 응답 |
| `backend` | `Up` 상태, `/api/posts`가 `200` 응답 |
| `postgres` | `Healthy` 상태 |
| `certbot` | `Up` 상태 |

홈 화면만 `200`이어도 backend는 장애 상태일 수 있다. React 정적 파일은 Nginx가 직접 제공하기 때문에 API 요청을 별도로 확인한다.

### 4.2 컨테이너 로그

```bash
docker compose --env-file .env.prod logs --tail 100 backend
docker compose --env-file .env.prod logs --tail 100 nginx certbot postgres
docker compose --env-file .env.prod logs -f backend
```

Compose의 컨테이너 로그는 `json-file` driver로 저장되고 컨테이너별 다음 제한으로 회전한다.

```text
최대 파일 크기: 10 MB
보관 파일 수: 3개
```

즉 컨테이너 파일 로그는 무제한 누적되지 않지만, 장애 분석에 필요한 오래된 로그를 장기 보존하는 구조도 아니다.

### 4.3 Nginx 요청 제한

Nginx는 클라이언트 IP 기준 요청 제한을 적용한다. 정상적인 화면 로딩과 관리자 사용은 허용하되, 반복 API 호출과 인증·메일·댓글 남용을 빠르게 차단하기 위한 1차 방어다.

| 대상 | 제한 | 초과 응답 |
| --- | --- | --- |
| HTTP->HTTPS redirect, React 페이지, 업로드 이미지 파일 | IP당 초당 20건, 순간 허용 40건 | `429 Too Many Requests` |
| `/api/` 전체 | IP당 초당 10건, 순간 허용 20건 | `429 Too Many Requests` |
| 관리자 로그인, 구독/구독취소 요청, 댓글 작성 | IP당 분당 5건, 순간 허용 2건 | `429 Too Many Requests` |
| Let's Encrypt ACME challenge | 제한 없음 | 인증서 발급/갱신에 사용 |

관련 설정은 [`tech-log-front/nginx.conf`](../../tech-log-front/nginx.conf)에 있다. 해당 제한은 단일 IP의 반복 요청 완화 용도이며, 다수 IP가 참여하는 대규모 DDoS나 AWS 네트워크 비용을 보장해서 막는 장치는 아니다. 공개 운영 규모가 커지거나 공격 비용 통제가 중요해지면 CloudFront와 AWS WAF rate-based rule, AWS Budgets 경보를 별도로 검토한다.

배포 후 Nginx 설정과 제한 응답을 점검한다.

```bash
cd /home/ubuntu/tech-log
docker exec tech-log-nginx nginx -t
docker exec tech-log-nginx nginx -T 2>&1 | grep -E 'limit_req|sensitive_request_ip'

for i in $(seq 1 25); do
  curl -s -o /dev/null -w '%{http_code}\n' 'https://techlog.site/api/posts?page=0&size=1'
done
```

마지막 반복 요청 중 일부가 `429`이면 API 제한이 동작하는 것이다. 운영 점검 중에는 같은 IP에서 잠시 API 요청이 제한될 수 있으므로 관리자 작업 중에는 실행하지 않는다.

## 5. DB에 저장되는 데이터와 삭제 정책

### 5.1 자동 삭제되는 데이터

Spring Boot에는 [`AuditLogRetentionService`](../../tech-log-back/src/main/java/com/nippyclouding/tech_log_back/log/AuditLogRetentionService.java)가 있으며, 기본적으로 매일 오전 `03:15` (`Asia/Seoul`)에 90일보다 오래된 감사 로그를 삭제한다.

| 테이블 | 내용 | 기본 보존 기간 | 자동 삭제 |
| --- | --- | ---: | --- |
| `ACCESS_LOGS` | 관리자 변경 요청, 댓글 작성 요청, 관리자 API 권한 오류, 서버 `5xx` 오류와 request ID/오류 상세 | 90일 | 예 |
| `LOGIN_LOGS` | 관리자 로그인 및 GitHub 로그인 성공/실패 기록, IP | 90일 | 예 |

설정은 EC2 `.env.prod`에서 변경할 수 있다.

```env
AUDIT_LOG_RETENTION_DAYS=90
AUDIT_LOG_CLEANUP_CRON=0 15 3 * * *
```

예를 들어 IP와 로그인 식별자 보관 기간을 30일로 낮추려면:

```env
AUDIT_LOG_RETENTION_DAYS=30
```

환경변수를 바꾼 뒤에는 backend를 다시 생성한다.

```bash
docker compose --env-file .env.prod up -d --no-deps --force-recreate backend
```

### 5.2 감사 로그 보존 스케줄러가 삭제하지 않는 데이터

아래 데이터는 90일 감사 로그 정리 작업의 대상이 아니다. 다만 게시글 삭제, 이미지 교체, 구독 취소처럼 서비스 기능을 직접 실행하면 그 기능의 정책에 따라 변경되거나 삭제될 수 있다.

| 데이터 | 저장 위치/테이블 | 자동 삭제 | 운영 조치 |
| --- | --- | --- | --- |
| 게시글 | `BOARDS` | 아니오 | 관리자 삭제 전까지 유지 |
| 카테고리 | `CATEGORIES`, `BOARD_CATEGORY` | 아니오 | 관리자 기능으로 관리 |
| 관리자 계정 | `ADMINS` | 아니오 | 활성/잠금 상태와 인증 정보를 운영자가 관리 |
| 댓글 | `COMMENTS` | 아니오 | 관리자 삭제 시 행을 제거하지 않고 삭제 상태로 표시 |
| 이메일 구독자와 토큰 | `NEWSLETTER_SUBSCRIPTIONS` | 아니오 | 구독 취소 시 해당 구독 제거, 장기 미확인 구독 정리 스케줄은 없음 |
| 이미지 메타데이터 | `IMAGES` | 아니오 | 게시글 수정/삭제 시 더 이상 사용하지 않는 이미지 연결은 삭제 |
| 업로드 이미지 파일 | Docker volume `tech-log-images` | 정기 삭제 없음 | 게시글 수정/삭제 시 불필요 파일은 삭제하며, 별도 주기적 orphan 정리 없음 |
| PostgreSQL 전체 데이터 | Docker volume `tech-log-postgres-data` | 아니오 | 정기 백업 필요 |
| TLS 인증서 | `data/certbot/conf/` | Certbot이 관리 | 백업 및 갱신 확인 필요 |
| `.env.prod` | EC2 파일 | 아니오 | 비밀값 포함, 안전하게 백업 필요 |

감사 로그 스케줄러는 게시글, 댓글, 구독자, 이미지 파일을 정리하지 않는다. 이들의 변경 또는 삭제는 각 관리 기능의 실행 결과다.

## 6. 관리자 로그 화면과 오류 추적

관리자 콘솔의 `로그` 탭에서는 다음을 조회할 수 있다.

| 화면 | 표시 정보 |
| --- | --- |
| 접근 로그 | IP, HTTP method, path, status, request ID, 서버 오류 타입/메시지/stack trace, 발생 시각 |
| 로그인 이력 | Provider/결과, login ID, IP, 발생 시각 |

서버 `500` 오류의 상세 정보는 새 오류가 발생한 시점부터 `ACCESS_LOGS`에 저장된다. 이전에 기록된 로그는 상세 필드가 비어 있을 수 있다.

DB 로그 저장이 실패하더라도 사용자 요청 자체를 추가로 실패시키지 않도록 구현되어 있다. 이 경우 backend 컨테이너 로그에서 감사 로그 저장 실패 메시지를 확인한다.

## 7. 배포 운영

### 7.1 자동 배포 흐름

`main` 브랜치 push 후 GitHub Actions는 다음을 수행한다.

```text
frontend lint/build + backend test
-> nginx/backend Docker 이미지 build
-> GHCR에 커밋 SHA 및 latest 태그 push
-> SSH로 EC2 접속
-> 대상 커밋 checkout
-> deploy/deploy.sh 실행
-> 해당 SHA 이미지 pull 및 컨테이너 갱신
```

GitHub Repository Actions variable에는 다음 값이 있어야 자동 배포된다.

```text
DEPLOY_ENABLED=true
```

이 값은 `production` Environment variable이 아니라 Repository Actions variable이어야 한다.

### 7.2 배포 후 확인

```bash
cd /home/ubuntu/tech-log
git rev-parse --short HEAD
docker inspect tech-log-backend --format '{{.Config.Image}}'
docker compose --env-file .env.prod ps
curl -I https://techlog.site
curl -i 'https://techlog.site/api/posts?page=0&size=1'
```

EC2의 실행 커밋과 컨테이너 이미지 태그는 배포한 GitHub 커밋 SHA와 일치해야 한다.

### 7.3 DB schema 변경 시 주의

`ddl.sql`과 `index.sql`은 PostgreSQL Docker volume이 비어 있는 최초 DB 초기화 시점에만 자동 실행된다. 기존 운영 DB에서 파일을 수정하고 컨테이너를 재시작해도 schema가 바뀌지 않는다.

현재는 Flyway를 도입하지 않았으며, 현재 필요한 schema 추가는 아래 migration들을 `deploy.sh`가 반복 실행 가능한 방식으로 적용한다.

```text
deploy/migrations/V20260527__access_log_error_details.sql
deploy/migrations/V20260527__admins.sql
```

새 테이블 또는 컬럼을 추가하려면 별도 migration 적용 방식이 반드시 필요하다. 변경이 더 늘어나기 전에 Flyway baseline 전환을 검토한다.

## 8. HTTPS 인증서 운영

현재 인증서는 Let's Encrypt로 발급되었고 최초 확인 만료일은 `2026-08-24`이다. Certbot 컨테이너는 12시간마다 갱신 필요 여부를 확인하고, Nginx는 6시간마다 reload하여 갱신 인증서를 읽는다.

월 1회 및 만료 약 30일 전에 확인한다.

```bash
cd /home/ubuntu/tech-log
echo | openssl s_client -connect techlog.site:443 -servername techlog.site 2>/dev/null | openssl x509 -noout -issuer -subject -dates
docker compose --env-file .env.prod run --rm --entrypoint certbot certbot certificates
docker compose --env-file .env.prod run --rm --entrypoint certbot certbot renew --dry-run
docker compose --env-file .env.prod logs --tail 100 certbot nginx
```

갱신이 실패하면 먼저 DNS, 보안 그룹의 `80`/`443`, ACME webroot 응답을 확인한다. 인증서가 이미 있는데 배포가 인증서 부재로 실패할 때에는 재발급부터 하지 말고 Certbot 컨테이너 내부 파일 존재와 `.env.prod`의 `DOMAIN` 값을 검사한다.

## 9. Gmail SMTP 운영

구독 메일은 Gmail API가 아니라 Gmail SMTP로 발송한다.

```env
MAIL_ENABLED=true
MAIL_HOST=smtp.gmail.com
MAIL_PORT=587
MAIL_USERNAME=nippyclouding@gmail.com
MAIL_PASSWORD=<Google 앱 비밀번호>
MAIL_FROM=nippyclouding@gmail.com
MAIL_SITE_NAME=Tech Log
```

`MAIL_PASSWORD`는 Google 계정 일반 비밀번호가 아니라 2단계 인증 후 발급한 앱 비밀번호여야 하며 Git에 커밋하지 않는다.

동작 확인 순서:

1. 공개 사이트에서 테스트 이메일로 구독을 신청한다.
2. 구독 확인 메일이 도착하는지 확인한다.
3. 확인 링크가 사이트로 redirect되어 구독 완료가 표시되는지 확인한다.
4. 테스트 게시글 발행 시 확인된 구독자에게 신규 게시글 메일이 전달되는지 확인한다.

발송 실패는 backend 로그에서 확인한다.

```bash
docker compose --env-file .env.prod logs --tail 150 backend
```

## 10. 백업과 복구 책임

현재 DB와 업로드 이미지는 같은 EC2의 EBS 위에 있으므로, EC2 또는 볼륨 손실 시 함께 손실될 수 있다. RDS나 S3의 자동 내구성을 현재 구성에서는 제공받지 않는다.

최소 백업 대상:

| 대상 | 위치 | 권장 처리 |
| --- | --- | --- |
| PostgreSQL | `tech-log-postgres-data` | `pg_dump` 정기 생성 후 EC2 외부 저장 |
| 업로드 이미지 | `tech-log-images` | EC2 외부 백업 또는 S3 이전 |
| 인증서 | `data/certbot/conf/` | 복구용 백업 |
| 운영 설정 | `.env.prod` | 비밀관리 방식으로 별도 보관 |

큰 배포, 인스턴스 타입 변경, DB migration 전에 최소한 DB dump 또는 EBS snapshot을 만든다.

## 11. 보안과 비용 관리

현재는 EC2에 Nginx가 직접 공개되어 있다. AWS Shield Standard의 기본 보호는 적용되지만, CloudFront/WAF 기반 HTTP 요청 rate limiting이나 원본 EC2 보호 구성은 아직 도입하지 않았다.

운영자가 즉시 유지해야 할 항목:

| 항목 | 현재/권장 상태 |
| --- | --- |
| 보안 그룹 `80`, `443` | 공개 서비스용 허용 |
| 보안 그룹 `8080`, `5432` | 외부 공개 금지 |
| SSH `22` | 배포 요구 범위 안에서 최소 허용, 전체 공개 상시 유지 금지 |
| GitHub/SMTP/DB 비밀값 | `.env.prod`와 GitHub Secrets 외 노출 금지 |
| AWS Budgets/Cost Anomaly Detection | 비용 사고 탐지를 위해 설정 권장 |
| CloudFront + WAF | 트래픽 증가 또는 공격 대응이 필요할 때 우선 검토 |
| S3 이미지 저장 | 이미지 백업·캐싱·EC2 부하 완화를 위해 향후 검토 |

## 12. 아직 도입하지 않은 운영 기능

다음 항목은 검토했으나 현재 코드 또는 AWS 구성에 적용된 것으로 가정하면 안 된다.

| 기능 | 현재 상태 |
| --- | --- |
| RDS | 사용하지 않음 |
| S3 이미지 저장 / CloudFront 캐싱 | 사용하지 않음, 이미지는 EC2 Docker volume 저장 |
| CloudWatch Logs / Agent / Alarm | 별도 설정 확인 전에는 미도입 상태로 본다 |
| Prometheus / Grafana | 사용하지 않음 |
| ELK / OpenSearch 로그 분석 | 사용하지 않음 |
| Flyway | 사용하지 않음, 현재 migration은 배포 스크립트가 직접 실행 |
| 관리자 계정 관리 UI | 사용하지 않음, `ADMINS` 기반 로그인만 구현 |
